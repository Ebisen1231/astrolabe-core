# M3.5 キックオフ指示(Codexへの発注プロンプト)

M4の前に挿入する**学習体験の磨き(Notion風の読書面)**。以下をそのままCodexに貼り付けて使う。

---

astrolabe-core の AGENTS.md、docs/design.md、**docs/ui-research.md** を読んでから作業を
始めてください。ui-research.md が本便の仕様源です(施主承認済みの採否マトリクス)。

**M3.5: 学習体験の磨き**を実装してください。M3(デプロイ+認証)は受け入れ済み。
目的は「読む・辿る・記録する」の摩擦を減らすこと。完了条件は
「**毎朝の報告が白地のブロック構造で読め、概念を辿れ、実践課題がワンクリックで
タスクになる**」。

## デザイン原則(施主承認済み)

- **「観測は夜、学習は昼」** — 星図マップとナビ枠は濃紺(#0B1226系)を維持し、
  それ以外の全ページを白地の読書面にする
- ライトトークン: 紙 #FFFFFF(面)/#FBFAF7(下地)、墨 #1F2328、補助 #6B6860、
  コールアウト地 #FBF7EC、アクセント #B08A2E(金環の血統)、リンク #2E6B8A。
  本文16px・行間1.75。CSS変数化してダーク面のトークンと同居させる

## 実装項目(ui-research.md の Tier 1)

1. **ライトテーマ移行** — マップ以外の全ページ(報告・詳細・履歴・タスク・チューター)
2. **サイドバー+ページツリー** — 今日の報告 / 履歴(日付) / 概念 / タスク / チューター。
   折りたたみ可。モバイルではドロワー化。上部タブは廃止
3. **トピック詳細ページ** — ブロック構造: パンくず/種別・時間チップ/H1/
   「なぜ今」コールアウト/summary段落/学習コンテンツ箇条書き/**実践課題カード+
   「タスク化」ボタン**(既存 POST /v1/tasks 系のcreate_task経路を呼ぶ)/関連チップ/出典/
   フィードバック4種。報告ページはカード一覧→クリックで詳細へ
4. **core: practice_task の分離** — 統合報告の構造化出力スキーマに実践課題を独立
   フィールドとして追加し、export契約へ反映。**契約変更のバージョン運用(schema_version
   を上げるか、後方互換の追加フィールドとするか)を計画で提示**すること。fixturesも更新
5. **概念ページ+バックリンク** — 星図ノードクリックの遷移先。summary・status・
   confidence・出典に加え、**この概念が登場した報告(日付リンク)・関連タスク・
   隣接概念**を一覧表示。バックリンク索引をexport時にcore側で生成するか、UI側で
   既存map/reportsからjoinするかは計画で比較提案
6. **⌘K クイック検索** — 概念・報告・タスクの横断検索(クライアントサイドで可)。
   **ヒットなし時は「チューターに聞く」導線**を出し、入力文字列を添えて /tutor へ遷移
7. **KaTeX数式レンダリング** — 学習コンテンツ内の数式を組版。**KaTeXの依存追加は
   施主承認済み**(それ以外の追加依存は従来どおり提案制)
8. **タスクビュー強化** — リスト+状態/種別フィルタ。evidence付き完了は維持
9. **コントラストのテスト固定** — 読書面の主要な文字/背景の組にWCAG AA(4.5:1)を
   テストで担保する

## 引き継ぐ制約

- 新ページもすべてAuthGate内(認証境界を壊さない)。fixtureモードのフォールバックと
  バナーは新ページでも機能させる
- 実データ・Secretsをリポジトリにコミットしない。LLMにHTML/UIを書かせない
- core側はM0承認条件を維持(pytestオフライン全緑ほか)。UIは vitest / lint /
  production build 緑
- スマホ幅での表示崩れがないこと(M3でモバイル利用が始まっている)

## 受け入れ条件

- fixtureモードで新UI全ページが環境変数なしで起動する
- 実データで: サイドバー遷移/トピック詳細のブロック表示/**タスク化ボタンで実台帳に
  taskとtask_createdイベントが生まれる**/概念ページにバックリンクが並ぶ/⌘Kで検索・
  未ヒット→チューター誘導/数式が組版される
- コントラストAAテストが緑
- 回帰なし: 星図・チューター・朝ジョブ・認証・既存フィードバック経路
- 受け入れ後にProductionデプロイ(従来どおり手動 `--prod`)し、施主がスマホ実機で
  読書面の可読性を確認する

まず実装計画(ページ/ルーティング構成・practice_task契約の扱い・バックリンク索引の
方式・コミット分割)を短く提示して施主の承認を待つこと。承認後に実装し、受け入れ
条件を自分で全部実行して報告する。AGENTS.mdのフェーズ表記はM3.5へ同期し、
design.mdの変更履歴に「M3.5を挿入(M4はその後)」を追記する。
