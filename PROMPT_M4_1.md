# M4 第1便 キックオフ指示(Codexへの発注プロンプト)

M4は2分割: **①復習間隔+週次自己改修(本便、core中心)** → ②PWA+Webプッシュ(UI中心)。
以下をそのままCodexに貼り付けて使う。

---

astrolabe-core の AGENTS.md と docs/design.md を読んでから作業を始めてください。

**M4第1便: 復習間隔と週次自己改修ジョブ**を実装してください。M3.5は受け入れ済み。
M4全体の完了条件は「**アプリの改修タスクが学習タスクとして流れてくる**」— 本便がその本体です。

## 確定済みの決定(施主承認 2026-07-20)

- **復習方式: 簡易な決定的スケジューラ(SM-2系)で開始**(§12の未決事項を解消)。
  ただしイベント設計はFSRS互換にし、将来の載せ替えは「関数差し替え+再導出」で済む形にする
- **週次自己改修の自動化は「発注書ドラフトまで」**。Codex Pro認証をActionsで回すことはしない
  (§11-4の規約リスク回避)。実装発注は従来どおり施主が手動でCodexへ
- 改修提案のissueは **private の astrolabe-ledger に作成**する(GITHub TOKENの範囲内で済み、
  学習内容がpublicに漏れない)。coreにissueは作らない

## 実装項目

### A. 復習間隔(スペースドリピティション)

1. **スケジューラ純関数** — `review_schedule(events, today)` を導出関数として実装
   (concepts/edgesの導出と同格の扱い)。quiz_result / marked_known / task_done /
   selected から各learned・review概念の次回復習期日と間隔を決定的に計算する。
   SM-2系の間隔遷移(グレードで伸縮)。パラメータはコード内定数。**同一イベント列→
   同一期日を含むpytestを必ず書く**
2. **quiz_result のグレード記録** — チューターのクイズ採点時、payloadに
   グレード(1=again / 2=hard / 3=good / 4=easy 相当)を追加記録(FSRS互換の布石)。
   既存の score(0..1)と derive の遷移規則は変えない(後方互換)
3. **朝の報告に「今日の復習」** — 期日到来の概念(上限5件)をトピックの後に掲載。
   テキスト報告・HTML・Discord embed・ReportExportV1(additive・schema_version 1据え置き)
   へ一貫反映。該当なしの日はセクション自体を出さない。UI画面への表示は第2便で行う
   (エクスポートに追加フィールドが入っても現行UIは無視できる)
4. **チューターの復習出題** — 「復習しよう」等の依頼で期日到来の概念からクイズを出し、
   採点時にグレード付き quiz_result を記録するツール拡張

### B. 週次自己改修ジョブ

5. **`astrolabe weekly-review`** — 直近7日の台帳(dismissedの傾向・chat_note・
   完了/未完了タスク・llm_usage実測・報告メタ)を flagship で分析(構造化出力)し、
   改修提案1〜3件を生成。それぞれについて:
   - **ledgerリポジトリにissue作成**(ラベル `self-improvement`。本文はCodexへ
     そのまま貼れる発注プロンプト形式: 背景・要求・受け入れ条件)
   - **台帳に `build_app_feature` タスク + task_created イベントを追記**
     (既存のcreate_task経路。これで「改修タスクが学習タスクとして流れてくる」)
   - Discord通知(提案の要約+issueリンク)
6. **プライバシーの縛り** — 提案文には学習内容の生データ(概念名の羅列や面談内容)を
   引用せず、機能面の不満・傾向の要約に留めるようプロンプトで指示し、テストで
   プロンプト文言を固定する(§2原則5。issueはprivateだが多層防御として)
7. **週次スケジュール** — ledger workflowに週1回のジョブを追加(曜日・時刻は
   朝ジョブと衝突しない提案をすること。workflow_dispatch併設・concurrency設定)。
   coreの参照コピーも同期。usage は llm_usage へ記録し、既存の予算ガードに従う

## 引き継ぐ制約

- M0承認条件を維持(pytestオフライン全緑・予算二重確認・リトライ全体1回・追記専用)
- 契約変更はすべて additive(schema_version 1 据え置き、決定性テスト更新)
- 新規Python依存なし想定(issue作成は既存のGITHUB_TOKEN+httpxパターン)。
  必要が生じたら実装前に提案
- `--date` 開発バイパスで復習期日を踏む多日シミュレートがテストできること
  (開発台帳のみ、本番台帳に合成データ禁止)

## 受け入れ条件

- スケジューラのpytest(決定性・グレード別の間隔遷移・期日到来判定)全緑
- 開発台帳で `--date` を進めながら「クイズ→グレード記録→期日再計算→朝の報告に
  『今日の復習』出現」の多日サイクルが動く
- 実台帳+実APIで `weekly-review` を1回実行し、ledgerのissue・build_app_feature
  タスク・Discord通知を確認(この初回実行が発生させる改修提案が、M4完了条件の実証になる)
- workflow_dispatch で週次ジョブが完走する
- 朝ジョブ・チューター・UI・既存経路に回帰なし。core/UIの全テスト・lint緑

まず実装計画(スケジューラの遷移規則・weekly分析の構造化出力スキーマ・週次workflowの
時刻設計・コミット分割)を短く提示して施主の承認を待つこと。承認後に実装し、
受け入れ条件を自分で全部実行して報告する。AGENTS.mdのフェーズをM4へ同期し、
design.mdの変更履歴に「M4開始・復習方式=簡易スケジューラ(FSRS互換イベント)・
週次自己改修は発注書ドラフトまで自動化」を追記する。
